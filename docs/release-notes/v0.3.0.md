# SwiftIdempotency 0.3.0

Third public release. Ships **Option B** — a test-helper surface for
effect-observation idempotency checking — addressing the limitation
the v0.2.0 README called out explicitly: `#assertIdempotent`
(Option C) is blind on handlers whose return type doesn't reflect
their side effects (`HTTPStatus.ok`, `Bool`, `Void`, etc.).

Additive; no breaking changes from v0.2.0.

## New

### `IdempotentEffectRecorder` protocol (in `SwiftIdempotency`)

A protocol that test doubles conform to so the helper can observe
their mutations. Lives in the main `SwiftIdempotency` target — not
`SwiftIdempotencyTestSupport` — so production instrumentation (retry
observability, metrics shims) can conform without pulling in the
test-support dependency.

```swift
import SwiftIdempotency

final class MockOrderRepository: OrderRepository, IdempotentEffectRecorder {
    private(set) var effectCount = 0
    private(set) var inserted: [Order] = []

    func insert(_ order: Order) async throws {
        inserted.append(order)
        effectCount += 1
    }
}
```

The protocol has an associated `Snapshot: Equatable` that defaults to
`Int` (backed by `effectCount`). Adopters can override with any
`Equatable` type — an ordered call log, a per-op counter dictionary,
whatever captures the state their idempotency contract rests on — to
detect non-idempotency invisible to counts alone (retries that
re-order ops, leave count unchanged but call order diverged).

### `assertIdempotentEffects(recorders:body:)` (in `SwiftIdempotencyTestSupport`)

A helper that runs a body twice and asserts every recorder's snapshot
is unchanged across the second invocation. Complements
`#assertIdempotent`'s Option C (return-equality) shape.

```swift
import SwiftIdempotencyTestSupport

@Test("addUser is idempotent when keyed on the request id")
func addUserIsIdempotent() async throws {
    let coinRepo = MockCoinRepo()
    let handler = UsersHandler(coinRepo: coinRepo)

    try await assertIdempotentEffects(recorders: [coinRepo]) {
        _ = try await handler.handleAddUser(entry: request)
    }
    // Passes iff the second invocation's snapshot equals the first —
    // coinRepo saw zero new puts on retry.
}
```

### `IdempotencyFailureMode` enum (in `SwiftIdempotency`)

New parameter on `assertIdempotentEffects` controlling how detected
non-idempotency is reported:

- `.preconditionFailure` (default) — `Swift.preconditionFailure(_:file:line:)`.
  Aborts the process; matches `#assertIdempotent`'s behavior and
  works outside a Swift Testing context.
- `.issueRecord` — `Testing.Issue.record(_:sourceLocation:)`. Fails
  the enclosing `@Test` without aborting, so failure-path tests can
  be exercised via `withKnownIssue { }`.

```swift
await withKnownIssue {
    await assertIdempotentEffects(
        recorders: [mockDB],
        failureMode: .issueRecord
    ) {
        await nonIdempotentHandler.run()
    }
}
```

## Changed

### `SwiftIdempotencyTestSupport` is no longer a placeholder

The target now ships real content — `assertIdempotentEffects` and
its internal imports of `Testing`. Adopters who want Option B must
add the product to their test target's dependencies; adopters who
use only `#assertIdempotent` (Option C) still don't need it because
the freestanding macro's runtime helpers live in `SwiftIdempotency`.

```swift
// Before (v0.2.x): SwiftIdempotencyTestSupport was a placeholder;
// adopters could omit the product entirely.

// After (v0.3.0): add the product if using assertIdempotentEffects.
.testTarget(
    name: "YourAppTests",
    dependencies: [
        .product(name: "SwiftIdempotency", package: "SwiftIdempotency"),
        .product(name: "SwiftIdempotencyTestSupport", package: "SwiftIdempotency"),
    ]
)
```

### README — "Effect-observation testing via `IdempotentEffectRecorder`"

New subsection #4 under "What it provides", paralleling the existing
`#assertIdempotent`, `@IdempotencyTests`, and `IdempotencyKey`
sections. Covers when to reach for Option B vs Option C, the protocol
conformance shape, `failureMode` selection, and the `Snapshot`
associatedtype with a `[String]` call-log example.

The forward-looking sentence in the `#assertIdempotent` limitations
subsection ("A future release may add a dependency-injected-effect
variant…") is now resolved and points at the new section.

## Validation

- **Root test suite:** 74 tests / 8 suites passing (64 from v0.2.0
  + 10 new in `AssertIdempotentEffectsTests`: R1 `.issueRecord` path
  × 2, R2 custom `Snapshot` × 2, default `Snapshot == Int` sanity × 1,
  existing prototype tests × 5).
- **Example samples:** 6 of 6 `examples/` packages green
  (webhook-handler 3, idempotency-tests 5, assert-idempotent 6,
  swiftdata 8, fluent 4, **option-b 3**). The new `option-b-sample`
  mirrors the existing sample convention (path dep on `../..`,
  matching the other consumer samples).
- **Real-adopter validation:** the Option B API shape was validated
  end-to-end in the Penny bot package-integration trial
  ([`Joseph-Cursio/penny-bot-idempotency-trial@c121ced`](https://github.com/Joseph-Cursio/penny-bot-idempotency-trial/tree/package-integration-trial)).
  The trial exercised the prototype on a real coin-double-grant bug
  shape (2/2 tests green on Penny's heavy dep graph: DiscordBM +
  Soto + OpenAPI + FluentKit + NIO). Findings informed the three
  pre-ship refinements: R1 (`failureMode`), R2 (`Snapshot`
  associatedtype), R3 (protocol home moved from TestSupport to
  main target). See
  [`docs/penny-package-trial/`](../penny-package-trial/) for the
  full trial artifacts.

## Migration from v0.2.0

**No action required** for existing v0.2.0 adopters. All prior
symbols — `IdempotencyKey`, the four attribute macros,
`@IdempotencyTests`, `#assertIdempotent`,
`IdempotencyKey(fromFluentModel:)` — are unchanged.

**Recommended adoption path for handlers with trivial returns:**

1. Extract a repo/service protocol behind the handler's
   persistence boundary (typically ~3-8 LOC per repo).
2. Write a mock conforming to both that protocol and
   `IdempotentEffectRecorder` (~20-30 LOC per mock; reusable across
   all tests touching the same repo).
3. Add `assertIdempotentEffects(recorders: [mock]) { ... }` call
   sites to existing `@Test` methods.

Total adopter-side cost measured against the Penny bot trial:
~35-50 LOC for a full repo-and-service extraction (the bulk is the
mock implementations, which amortise across the whole test suite).
See the Penny retrospective's "refactor cost summary" for per-layer
LOC breakdown.

## What's not here (still deferred to future releases)

- **Macro-wrapping variant** (`#assertIdempotentEffects` as a
  freestanding expression macro) — the free function is clearer for
  the shape the trial validated; macro variant can follow if
  adopters ask.
- **Hybrid Option-B/Option-C** (`assertIdempotent(returning:,
  effects:)`) — adopters needing both checks can compose two
  separate calls.
- **Parameterised `@IdempotencyTests` expansion** — zero-argument
  members only still.
- **Penny cross-bug-shape coverage trial** — the v0.3.0 ship was
  validated against the coin-double-grant shape. A follow-up
  Penny-wide trial covering all four Penny bug shapes (OAuth,
  sponsor DM, GHHooks error-path, coin-double-grant) is a
  candidate Next Step if/when Option B hits friction in the wild.
