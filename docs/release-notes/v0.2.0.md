# SwiftIdempotency 0.2.0

Second public release. Additive Fluent-integration feature; no
breaking changes from v0.1.0.

## New

### `SwiftIdempotencyFluent` product

New opt-in library product pairing SwiftIdempotency with Fluent
`Model` types. Provides a dedicated initializer:

```swift
import SwiftIdempotency
import SwiftIdempotencyFluent

let key = try IdempotencyKey(fromFluentModel: savedAcronym)
// key.rawValue == savedAcronym.requireID().uuidString
```

Fluent adopters who previously had to write a per-Model `Identifiable`
adapter struct (bridging Fluent's missing `Identifiable` conformance
and force-unwrapping the Optional `UUID?` id) can now reach
`IdempotencyKey` directly from the bare Fluent Model. The
initializer throws `FluentError.idRequired` on pre-save nil ids,
matching FluentKit's own `requireID()` idiom.

### Package graph addition

```swift
// Adopter Package.swift
.package(url: "https://github.com/Joseph-Cursio/SwiftIdempotency.git",
         from: "0.2.0"),
// ...
.product(name: "SwiftIdempotency", package: "SwiftIdempotency"),
.product(name: "SwiftIdempotencyFluent", package: "SwiftIdempotency"),  // ← new
```

Non-Fluent adopters don't add the `SwiftIdempotencyFluent` product
and pay zero additional compile cost. The FluentKit dep is gated
on the opt-in product.

## Constraints

- **IDValue constraint:** `M.IDValue: CustomStringConvertible`.
  Covers `UUID`, `Int`, `String`, and typed wrappers. Models using
  `@CompositeID` (custom-struct IDValue) are rejected at compile
  time; route those through `IdempotencyKey(fromAuditedString:)`
  on a manually-composed string.
- **Post-save only:** pre-save Models with nil id throw
  `FluentError.idRequired`. For create handlers, continue using
  the header-sourced `init(fromAuditedString:)` pattern
  documented in the README's "Using with Fluent ORM" section.
- **FluentKit minimum:** `vapor/fluent-kit >= 1.48.0`. The
  `Model.requireID()` + `IDValue` associated-type API has been
  stable since Fluent 4, so the conservative floor is well ahead
  of the API surface used.

## Changed

### README — "Using with Fluent ORM"

New subsection "Post-save keys: `init(fromFluentModel:)` (v0.2.0+)"
documents the opt-in product and the direct-constructor pattern.
The pre-v0.2.0 `IdentifiableAcronym` adapter-struct pattern is
preserved below as a v0.1.x-compatible fallback. The upgrade
recommendation is to switch to `init(fromFluentModel:)`.

### `.spi.yml`

`documentation_targets` now includes both `SwiftIdempotency` and
`SwiftIdempotencyFluent` so SPI's DocC pipeline surfaces the new
initializer's documentation.

## Validation

- **Root test suite:** 64 tests / 7 suites passing (58 from v0.1.0
  + 6 new in `SwiftIdempotencyFluentTests`).
- **Example samples:** 5 of 5 `examples/` packages green
  (webhook-handler 3, idempotency-tests 5, assert-idempotent 6,
  swiftdata 8, **fluent 4**).
- **Real-adopter validation:** the `Joseph-Cursio/HelloVapor-idempotency-trial`
  fork's `package-integration-trial` branch was re-migrated to
  v0.2.0 as the pre-tag validation gate. 5 of 5 tests pass on
  the re-run; the `IdentifiableAcronym` adapter struct was
  removed (−9 LOC per Model keyed on); commit:
  [`Joseph-Cursio/HelloVapor-idempotency-trial@3c8a0f7`](https://github.com/Joseph-Cursio/HelloVapor-idempotency-trial/commit/3c8a0f7).

## Migration from v0.1.0

**No action required.** v0.1.0 adopters continue compiling without
change. The pre-v0.2.0 `IdentifiableAcronym`-adapter pattern still
works — the `init(fromEntity:)` generic constructor is unchanged.

**Recommended path for Fluent adopters:** add the
`SwiftIdempotencyFluent` product to your Package.swift, swap
adapter-struct call sites to `IdempotencyKey(fromFluentModel:)`,
and remove the adapter types. See the hellovapor re-run commit
for a worked migration on a real adopter.

## What's not here (still deferred to future releases)

- **Additional ORM integrations** (SwiftData already reaches
  `IdempotencyKey(fromEntity:)` cleanly — see the
  [synthetic-swiftdata trial](docs/synthetic-swiftdata-package-trial/trial-findings.md)
  — so no dedicated product needed there).
- **Option B (dep-injected mock effects)** for `#assertIdempotent`.
- **Parameterised `@IdempotencyTests` expansion** — zero-argument
  members only still.
- **Current-toolchain SwiftData real-adopter trial** — external-
  adoption signal gap remains open.
