# SwiftIdempotency 0.1.0

First public release. Compile-time type enforcement and test
scaffolding for idempotent operations in Swift.

## What it provides

Three tiers of safety against retry-duplication bugs:

1. **Compile-time type enforcement** via `IdempotencyKey`.
   A strong type with no `init()` — callers must go through
   `IdempotencyKey(fromEntity:)`, `IdempotencyKey(fromAuditedString:)`,
   or `IdempotencyKey(rawValue:)` to construct one. Using `UUID()`
   or similar per-invocation generators as an idempotency key
   becomes a type mismatch at compile time.
2. **Annotation macros** for lattice-aware effect declaration:
   - `@Idempotent` — re-invocation produces the same observable
     result and no additional external effects.
   - `@NonIdempotent` — re-invocation produces additional effects.
   - `@Observational` — only observation-primitive side effects
     (logging, metrics, spans); retry-safe by convention.
   - `@ExternallyIdempotent(by: "parameterName")` — idempotent
     when routed through the named deduplication-key parameter.
     The `by:` label identifies which parameter the
     `missingIdempotencyKey` linter rule verifies.
3. **Test scaffolding** for dynamic idempotency checks:
   - `@IdempotencyTests` — extension macro attached to a Swift
     Testing `@Suite` type; emits one `@Test` per `@Idempotent`
     zero-argument member.
   - `#assertIdempotent { ... }` — freestanding expression macro.
     Runs the closure twice and asserts equal return values. Sync
     and async overloads; throwing propagates through.

## Validation

Shipping state is validated across:

### Adopter road-tests (thirteen production Swift apps)

All thirteen rounds completed: todos-fluent, pointfreeco/www,
swift-nio, swift-composable-architecture, swift-aws-lambda-runtime,
penny-bot, isowords, SwiftPackageIndex-Server, prospero,
myfavquotes-api, hummingbird-examples/open-telemetry, luka-vapor,
hellovapor.

**Ten real-bug shapes across six production adopters**, all of
which map to `IdempotencyKey` + `@ExternallyIdempotent(by:)` —
see `docs/next_steps.md` for the cross-adopter table.

### Package-adoption trials (three distinct integration shapes)

- **luka-vapor** — Redis + APNS, `HTTPStatus.ok` return. Stressed
  the `#assertIdempotent` "trivial-return" pathology.
- **hellovapor** — Fluent `Acronym` Model, Optional `UUID?` ID.
  Surfaced the `IdempotencyKey(fromEntity:)` reachability
  constraint on Fluent Model types and the idiomatic workarounds
  (adapter struct, header-sourced `init(fromAuditedString:)`).
- **synthetic SwiftData** — `@Model final class` + reference-type
  `Identifiable`, both with non-Optional `id: String`. Confirmed
  that the `CustomStringConvertible` constraint gap is
  **Fluent-specific**, not a generalised issue.

### Consumer samples (four end-to-end SPM packages)

Under `examples/`:
- `webhook-handler-sample/` — `IdempotencyKey` type safety.
- `idempotency-tests-sample/` — `@IdempotencyTests` extension
  expansion on a `@Suite` type.
- `assert-idempotent-sample/` — `#assertIdempotent` across the
  sync/async × throwing/pure effect matrix.
- `swiftdata-sample/` — `@ExternallyIdempotent(by:)` +
  `IdempotencyKey(fromEntity:)` on a SwiftData `@Model` shape.

Each sample has passing tests:
- Root `SwiftIdempotencyTests`: 58 tests / 6 suites.
- `examples/`: 22 tests (swiftdata 8 + assert-idempotent 6 +
  idempotency-tests 5 + webhook-handler 3).

## Installation

```swift
// Package.swift
dependencies: [
    .package(url: "https://github.com/Joseph-Cursio/SwiftIdempotency.git",
             from: "0.1.0"),
],
targets: [
    .target(
        name: "YourApp",
        dependencies: [
            .product(name: "SwiftIdempotency", package: "SwiftIdempotency"),
        ]
    ),
]
```

Requirements: Swift 5.10+, Swift Testing (for `@IdempotencyTests`
expansion into `@Test` methods). Supports macOS 13+, iOS 16+,
tvOS 16+, watchOS 9+.

## Linter companion

Pairs with **[SwiftProjectLint](https://github.com/Joseph-Cursio/SwiftProjectLint)**,
which enforces the lattice at lint time. The linter recognises both
the doc-comment (`/// @lint.effect idempotent`) and attribute
(`@Idempotent`) forms identically. Rules that consume the macro
surface:

- `idempotencyViolation` — non-idempotent call from an
  `@Idempotent` / `@lint.context replayable` scope.
- `missingIdempotencyKey` — call site passes a per-invocation
  generator (UUID(), Date(), etc.) to a parameter marked by
  `@ExternallyIdempotent(by:)`.
- `actorReentrancyIdempotencyHazard` — structural AST check for
  actor methods whose body does `guard ... contains ... → await
  ... → insert into same stored property`.
- `effectVariesByBranch` — a function's branches join to
  incompatible effects.

Using the package without the linter is fully supported — see
README §"Using without SwiftProjectLint" for the type- and
test-scaffold-only subset.

## Known limitations

- **`@IdempotencyTests`** requires Swift Testing. XCTest-only
  adopters can use `@Idempotent` as a marker but won't get
  generated test methods.
- **`#assertIdempotent`** requires the closure's return type to
  conform to the `Equatable` *protocol*. Tuples of Equatable
  elements have synthesised `==` but do not conform to the
  protocol — use a dedicated Equatable `struct` projection
  when returning from a non-Equatable reference (e.g. Fluent
  `Model`, SwiftData `@Model`). See README §"Using with Fluent
  ORM" / §"`#assertIdempotent` on Model returns" for the pattern.
- **`IdempotencyKey(fromEntity:)`** requires the entity's
  `ID: CustomStringConvertible`. Most typed-wrapper UUIDs and
  plain Strings satisfy this; Fluent `Model` types with
  `@ID var id: UUID?` do not (Optional doesn't conform). See
  README for the `IdentifiableAcronym`-style adapter pattern
  and the header-sourced `init(fromAuditedString:)` alternative.
- **`SwiftIdempotencyTestSupport` library is a placeholder.**
  The freestanding `#assertIdempotent` macro's runtime helpers
  live in the main `SwiftIdempotency` target, so test targets
  only need the one product. The placeholder exists so future
  test-only helpers can ship without breaking adopter Package.swift
  files that declare it.

## What's not here (deferred to post-0.1.0)

- **Dedicated Fluent-shaped `IdempotencyKey` constructor** —
  `init(fromFluentModel:)` or `init(fromFluentID:)` to avoid the
  adapter-struct boilerplate. Design TBD; needs FluentKit
  real-adopter validation before merging.
- **Option B (dependency-injected mock effects)** for
  `#assertIdempotent`. Option C (return-equality) is what this
  release ships; Option B would catch shapes Option C is blind
  to (trivial returns, non-Equatable returns without workaround
  struct). Design surface constrained by the luka-vapor +
  hellovapor + synthetic trial characterisations.
- **Current-toolchain SwiftData real-adopter trial.** The
  synthetic closed the API question; the external-adoption
  signal for SwiftData is TBD on a future real-adopter run
  (candidates: TeymiaHabit, vreader).
