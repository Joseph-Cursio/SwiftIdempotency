# SwiftIdempotency 0.4.0

Fourth public release. Ships the opt-in **`SwiftIdempotencyPropertyBased`**
product — **property-based** (generated-input, shrinking) retry-idempotence
testing — addressing the limitation the existing
`PropertyBasedAssertIdempotentTests` called out: `#assertIdempotent` fails via
`precondition`, which terminates the process *before* a property-based
shrinker can run, so a failing property never gets minimized.

Additive; no breaking changes from v0.3.x. The new product is opt-in — only
test targets that depend on it pull in `swift-property-based`.

## New

### `SwiftIdempotencyPropertyBased` product

A new library target, mirroring the `SwiftIdempotencyFluent` opt-in pattern.
`swift-property-based` is a real dependency of *this target only*; the lean
core is untouched.

### `assertIdempotentProperty(over:)` — value-level

For each input drawn from a generator, runs the operation **twice** and asserts
the results are equal (retry-stability). On failure it records a `Testing`
issue (non-fatal) instead of crashing, so `swift-property-based`'s shrinker
reduces the failure to the **minimal failing input**.

```swift
import SwiftIdempotencyPropertyBased
import PropertyBased

@Test func upsertIsRetryIdempotent() async {
    let service = UserService()
    await assertIdempotentProperty(over: Gen<Int>.int(in: 0 ... 1_000)) { id in
        await service.upsert(id)
    }
}
```

This is also the **supported path for parameterised `@Idempotent` functions**:
`@IdempotencyTests` only auto-generates tests for zero-argument members; cover
functions that take arguments with `assertIdempotentProperty` over a generator
of those arguments.

### `assertIdempotentEffectsProperty(over:makeRun:)` — action-sequence, effect-level

The effect-counting analogue: generates *sequences* of actions, builds a fresh
system per trial (via `makeRun`, returning the `IdempotentEffectRecorder`s + an
`apply` closure), applies the sequence **twice**, and asserts the retry adds no
new effects. On failure the action sequence is shrunk to the **minimal sequence**
that breaks retry-safety — the seed of model-based testing.

```swift
@Test func handlerIsEffectIdempotentAcrossSequences() async {
    await assertIdempotentEffectsProperty(
        over: Gen<Int>.int(in: 0 ... 5).array(of: 0 ... 8)
    ) {
        let repo = MockRepo()
        let handler = Handler(repo: repo)
        return (recorders: [repo], apply: { id in await handler.upsert(id) })
    }
}
```

### Replaying a failure

Use `swift-property-based`'s own seed trait — `@Test(.fixedSeed("…")) func …` —
to re-run a failing property from the seed it reported.

## Migration from v0.3.x

**No action required.** All prior symbols are unchanged. To adopt the new
helpers, add the `SwiftIdempotencyPropertyBased` product to your test target
(it pulls in `swift-property-based`).

## What's not here (deferred / rejected)

- **A persisted corpus / replay DB for these property tests — rejected.**
  `swift-property-based`'s `FixedSeedTrait` (`.fixedSeed(_:)`) already replays a
  failing property from its seed; a custom corpus would duplicate it, can't
  capture the failing seed programmatically without forking the runner, and a
  cross-repo shared corpus with SwiftInferProperties is a wrong-direction
  dependency. See `docs/property-based-assertions-design.md` (W6.B).
- **`@IdempotencyTests(inputs:)` macro** — a macro can't synthesise a generator
  for an arbitrary parameter type, so the adopter would still supply one;
  deferred pending a concrete arg-provision design. Use `assertIdempotentProperty`.
